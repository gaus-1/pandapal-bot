# Настройка и восстановление бэкапов базы данных (PostgreSQL)

Этот файл описывает важные правила для работы с автоматическими бэкапами базы данных в GitHub Actions (`db-backup.yml`) и подключением из внешнего мира к базе в Railway.

## 1. Строка подключения для GitHub Actions (и pgAdmin)

Для того чтобы скрипт бэкапа или ваша локальная программа (pgAdmin, DBeaver) могла достучаться до базы, которая крутится внутри Railway, необходимо использовать **внешний (TCP) Proxy**.

Пример правильной строки подключения `DATABASE_PUBLIC_URL` для GitHub Secrets:
`postgresql://<USER>:<PASSWORD>@turntable.proxy.rlwy.net:<PORT>/<DATABASE>`

> **⚠️ КРИТИЧЕСКАЯ ОШИБКА (Частая проблема):**
> Когда вы копируете строку из Railway Dashboard (или пароль) и вставляете в GitHub Secrets `.env` или поля Railway, очень легко случайно захватить **невидимый символ переноса строки (`\n`)**.
>
> Из-за этого база данных выдаст ошибку `database "railway\n\t" does not exist`, а аутентификация пользователя будет постоянно падать, так как пароль вместе с нажатием "Enter" считается неправильным.
>
> **Решение:** Всегда кликайте в самый конец скопированной строки (в GitHub Secrets или Railway Variables) и нажимайте `Backspace`. Мигающий курсор должен стоять вплотную к последнему символу. Пустых строчек внизу быть не должно!

## 2. Проблема версий (pg_dump Version Mismatch)

Если в логах бэкапа (в GitHub Actions) появляется ошибка:
`pg_dump: error: aborting because of server version mismatch`
`pg_dump: detail: server version: 18.3; pg_dump version: 16.13`

Это означает, что вы обновили версию PostgreSQL на Railway (например, до v18), но контейнер, который скачивает бэкапы в GitHub Actions, всё ещё работает на старой версии `postgres:16`. Утилита `pg_dump` отказывается делать бэкапы более новых баз.

**Решение:**
Зайдите в файл `.github/workflows/db-backup.yml` и обновите версию контейнера до актуальной. Например:
```yaml
    container:
      image: postgres:18   # Версия должна быть >= версии сервера Railway
```

## 3. Как вручную запустить бэкап
1. Откройте вкладку **Actions** в GitHub-репозитории.
2. Слева в меню выберите **Database Backup**.
3. Справа нажмите кнопку **Run workflow**.
4. После успешного скачивания артефакт (сжатый файл базы данных `pandapal_backup_...sql.gz`) будет доступен в самом низу логов завершившегося экшена. Этот файл хранится 30 дней.
