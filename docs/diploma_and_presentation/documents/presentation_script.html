Сам ответ я стримлю клиенту по протоколу SSE (Server-Sent Events) напрямую из YandexGPT Pro порциями в реальном времени, что полностью исключает интерфейсные зависания».

❓ Возможные вопросы комиссии (FAQ)

Этот блок — ваша шпаргалка для ответов на вопросы преподавателей. 11 вопросов по бэкенду и 4 по фронтенду. Читать их на видео не нужно, держите перед глазами на защите.

🔥 Бэкенд и Архитектура (11 вопросов)

1. Как вы решили проблему "галлюцинаций" (выдуманных фактов) нейросети при ответах школьникам?
Ответ: Я внедрил архитектуру RAG (Retrieval-Augmented Generation). Бэкенд сначала векторизует (Embeddings) вопрос пользователя, затем через расширение pgvector и индекс HNSW ищет релевантный школьный материал в базе PostgreSQL по косинусному сходству. Нейросеть отвечает строго на базе найденного контекста, а не придумывает ответ из воздуха.

2. Почему для кэширования и стейтов тамагочи вы выбрали именно Redis (In-Memory БД), а не сохраняете всё в PostgreSQL?
Ответ: Игровые механики генерируют огромное количество мелких операций записи/чтения (I/O). Сохранение каждого клика в дисковую БД (PostgreSQL) убило бы производительность. В Redis данные хранятся в оперативной памяти (с TTL), давая мгновенный ответ (<2мс), а в Postgres я переношу только финальный, агрегированный результат.

3. Как именно технически устроена ваша 4-уровневая система безопасности детей? Почему бот блокирует мат, но может рассказать про виды исторического оружия?
Ответ: Я внедрил алгоритмы Context-Aware (контекстно-зависимой) модерации. Запрос проходит конвейер: регулярные выражения, дистанция Левенштейна (поиск обхода фильтров) и проверку по паттернам триггеров (оружие, политика, 18+). Но система умная: если эвристический анализатор определяет намерение (Intent) как "Образовательное" (например, доклад по истории), бэкенд математически снижает вес (Confidence) угрозы на 70%. В итоге безопасный учебный контекст пропускается к LLM, а реальная агрессия жестко блокируется.

4. Вы используете SSE (Server-Sent Events) для стриминга ответов. Почему не WebSockets?
Ответ: Для генерации текста нужен только однонаправленный поток (сервер → клиент). WebSockets — это двунаправленный, тяжеловесный протокол, требующий поддержки состояния (stateful). SSE работает поверх обычного HTTP, идеально масштабируется, встроен в браузерные API и потребляет минимум ресурсов.

5. Как обеспечивается безопасность банковских платежей при подписке на Premium?
Ответ: Банковские данные (номера карт) вообще не проходят через мой сервер — всё процессит ЮKassa на своей стороне по стандарту PCI DSS. Мой бэкенд выставляет только Webhook-эндпоинт на aiohttp. Когда оплата проходит, ЮKassa шлёт POST-запрос. Мой код валидирует IP-адрес отправителя и проверяет JWT-подпись, прежде чем начислить Premium пользователю в базу.

6. Вы назвали архитектуру "Асинхронный микромонолит с Dependency Injection". Что это дает на практике?
Ответ: Микромонолит означает, что код лежит в одном репозитории, но строго поделен на независимые домены (Сервисы, Репозитории, Роутеры). Dependency Injection (внедрение зависимостей) позволяет мне легко подменять модули. Например, при тестировании я подменяю реальный класс YandexGPT на Mock-класс (заглушку), не меняя ни строчки в бизнес-логике самого бота.

7. Как реализован Rate Limiting (защита от DDoS и спам-ботов)?
Ответ: Я внедрил алгоритм "Token Bucket" с использованием Redis. Каждому Telegram ID выдается условное количество "токенов" (лимит запросов в секунду). Если школьник начинает спамить, токены сгорают, бэкенд прерывает обработку (HTTP 429) и игнорирует запросы, не тратя ресурсы процессора и платные токены LLM.

8. Вы упомянули CI/CD и Alembic. Как автоматически применяются миграции базы данных при деплое?
Ответ: В проекте настроен пайплайн GitHub Actions. При деплое на Railway собирается новый Docker Image. Внутри `entrypoint.sh` скрипта (или Procfile) я прописал команду запуска миграций `alembic upgrade head`. Она выполняется строго до запуска самого веб-сервера aiohttp, гарантируя, что код всегда работает с актуальной структурой таблиц БД.

9. Как сервер технически обрабатывает фотографию тетради с домашним заданием?
Ответ: Бот получает от Telegram File ID изображения. Бэкенд скачивает картинку во временный байтовый буфер (без сохранения на диск), кодирует её в Base64 и отправляет асинхронный HTTP-запрос в Yandex Vision API (модуль OCR). Назад сервер получает спарсенный текст и передает его в цепочку генерации ответа.

10. Что такое Resilience-тесты (тесты на отказоустойчивость), которые вы написали?
Ответ: Это тесты, проверяющие, как система ведёт себя при авариях. В Pytest я симулирую падение Redis или недоступность API Яндекса (выкидываю таймаут-ошибки). Тест проверяет, что бэкенд не крашится (Fallback механика) и возвращает пользователю человекопонятное сообщение, например: "Сервис временно перегружен", а не выдает критическую ошибку 500.

11. Почему для базы PostgreSQL выбран именно драйвер psycopg3, а не стандартный psycopg2?
Ответ: Мой бэкенд (aiohttp, aiogram) — полностью асинхронный и работает на Event Loop (цикл событий). Старый psycopg2 — блокирующий драйвер (синхронный), он заморозил бы всё ядро сервера при долгом ответе от базы. А вот psycopg3 — полностью асинхронный драйвер под капотом. Он отпускает поток во время ожидания I/O операции (чтения/записи в БД), позволяя моему серверу держать тысячи соединений параллельно.

🖥 Фронтенд: Вебсайт и Mini App (4 вопроса)

12. Откуда бэкенд знает, что запрос из Mini App отправлен реальным пользователем, а не хакером?
Ответ: При открытии Mini App, Telegram передает на фронтенд строку initData. Фронтенд отправляет её бэкенду. Мой сервер берет секретный токен бота (из `.env`), хэширует строку алгоритмом HMAC-SHA-256 и сверяет с контрольной суммой от Telegram. Подделать этот хэш без токена бота невозможно.

13. Почему вы выбрали Zustand для фронтенда, а не классический Redux?
Ответ: В React 19 классический Redux избыточен: слишком много бойлерплейт-кода (actions/reducers). Zustand легковесный, использует хуки из коробки, не оборачивает всё приложение в Context Provider и идеально подходит по масштабу для SPA уровня Mini App, делая код чище и быстрее.

14. У вас используется TanStack Query. Зачем он нужен, если уже есть Zustand?
Ответ: Это кардинально разные вещи. Zustand управляет локальным UI-стейтом (тема, октрыто ли окно и т.д.). А TanStack Query — это инструмент работы с серверным стейтом. Он сам делает HTTP-запросы, кэширует их, делает ретраи (повторы) при ошибках сети и инвалидирует кэш без написания "костылей" на useEffect.

15. Вы упомянули Lazy Loading (ленивую загрузку). Какую техническую метрику это улучшает?
Ответ: Это улучшает LCP (Largest Contentful Paint) и TTI (Time to Interactive). Код фронтенда разбивается (Code Splitting) конфигурацией Vite на мелкие чанки (куски). При входе пользователь скачивает только код главной страницы. Код тяжёлого "Чата" или "Игр" скачивается из сети только в момент перехода на эти вкладки, экономя трафик.
